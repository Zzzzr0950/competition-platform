# 张三（组长）：后端和数据库

## 我做了什么

> 一句话：**我负责让系统能跑起来——数据库、登录、审核、上线。**

---

## 讲的顺序

### 第一步：数据库（40秒）

> "我们设计了四张表。"

- **users** — 存学生和管-理员账号，学号唯一
- **submissions** — 存每一条申报记录
- **competition_catalog** — 存125个竞赛的基本信息
- **submission_logs** — 存每一次审核操作，谁在什么时候做了什么

> "一个学生可以申报多个竞赛，一条申报对应一个竞赛。通过外键把四张表关联起来。"

---

### 第二步：登录和权限（30秒）

> "登录用学号加密码。密码存的是哈希值，用 scrypt 算法加密，数据库泄露也看不到明文。"

> "权限控制用装饰器——login_required 检查是否登录，admin_required 检查是不是管理员。没登录就跳回登录页，没权限就提示'无权访问'。"

---

### 第三步：审核流程（1分钟）

> "管理员打开审核页面，可以选择只看待审核的、只看某个班的、搜索学生姓名学号。点进去看到申报详情和证书图片，填写意见后点通过或驳回。"

> "多人同时审核也不怕——如果一条申报已经被别人审过了，你再点就会提示'已被其他人审核过了，请刷新'。这是加了 pending 状态检查。"

> "每次审核都会记日志——谁审的、什么时间、结果是啥、评语是啥。以后可以追查。"

---

### 第四步：上线部署（30秒）

> "部署在 PythonAnywhere 免费云平台。改完代码 push 到 GitHub，然后在 PythonAnywhere 上 git pull 再点 Reload 就更新了。"

---

## 可能被问的问题

**问：为什么用 SQLite 不用 MySQL？**

- 答：1257个学生，一年两千条申报，SQLite 完全够用
- SQLite 不用单独装数据库软件，零维护
- 以后如果数据量大了，换 MySQL 只需要改一行配置

**问：密码安全怎么保证？**

- 答：scrypt 算法加密存储，存的是哈希值不是明文
- 即使有人拿到数据库文件，也反推不出密码

**问：怎么防止 SQL 注入攻击？**

- 答：所有 SQL 查询都用 ? 占位符传参数，不拼字符串

**问：服务器重启了会不会所有人都要重新登录？**

- 答：不会。我们把 SECRET_KEY 存到文件里了，重启不会变

## 可能被问的问题

**问：为什么没用 Vue 或 React？**

- 答：总共就 10 个页面，逻辑不复杂
- 用框架要装一堆依赖，打包后文件大，加载慢
- 原生 JS 写出来就几十 KB，手机秒开
- 对于这个体量的项目，原生方案正好

**问：不同手机屏幕怎么适配？**

- 答：用百分比宽度和弹性布局，不用写死的像素值
- 在 iPhone SE 小屏、iPhone 14、安卓手机、iPad 上都测过

**问：页面加载快吗？**

- 答：很快。CSS 大概 10KB，JS 总共 20KB，没有大图标库
- 首屏不到 1 秒就能出来

**问：网络不好提交失败了怎么办？**

- 答：前端先检查一遍，不合格不发出请求
- 提交中按钮禁用防重复
- 失败了会弹红条提示具体原因

## 可能被问的问题

**问：为什么不用 ECharts 做图表？**

- 答：ECharts 很大，1MB 多。我们用 Canvas 原生画只需要 180 行代码
- 自己画的好处是完全可控，颜色渐变、字号自适应这些细节都能自己定
- 数据不多的情况下，轻量方案更合适

**问：Excel 单元格怎么合并的？**

- 答：先把同一个学生的申报归到一起。第一行写完整的学号姓名总学分，后面几行这些格子空着。然后用 openpyxl 的 merge_cells 把这几个空行并成一个大格。

**问：名称清理是怎么做的？**

- 答：找字符串里有没有 ①②③ 这些圈号数字。找到了就往前看有没有分隔符（破折号、冒号、顿号），从分隔符那里截断。没有圈号数字的就不处理。就是一个正则匹配加字符串截取。

**问：导入脚本重复跑会不会重复导入？**

- 答：不会。导入前先查数据库，学号已经在的跳过。可以放心重复跑。

**问：怎么确保一开始导入了正确的学生？**

- 答：导入完了会打印每个班导入了多少人。我们和原始 Excel 对照，总数对得上。

# 答辩常见问题 —— 额外 25 题

## 系统整体类

**Q1：这个系统和学校的教务系统有什么关系？**
> 答：目前是独立运行的系统，和教务系统没有直接对接。竞赛申报本身就不在教务系统的功能范围内，我们做的是填补这个空白。以后如果需要，可以通过 API 对接学分数据。

**Q2：系统上线后有多少人用过？反馈怎么样？**
> 答：目前已经导入了全院 1257 名学生，管理员已经开始使用审核和导出功能。反馈主要是申报流程比纸质方便很多，手机提交不用跑办公室。管理员觉得班级筛选导出最实用。

**Q3：如果系统挂了怎么办？数据会丢吗？**
> 答：PythonAnywhere 有自动备份。数据库是文件型的，可以直接复制 db 文件做冷备份。我们还有一个 students_export.sql 文件存着所有学生数据，出问题可以快速恢复。

**Q4：有没有考虑过数据隐私问题？**
> 答：考虑了。密码用 scrypt 加密存哈希值。只有管理员能看到学生数据，学生之间看不到彼此的申报。没有对外暴露任何数据查询接口。

**Q5：这个系统最大的亮点是什么？**
> 答：全流程数字化——从申报到审核到统计到导出，一条龙。而且零经费、开源、可复制。别的学院拿过去换一下竞赛目录和学生名单就能用。

## 技术选型类

**Q6：为什么选 Python + Flask？不用 Java 或 PHP？**
> 答：Python 简单，开发快。Flask 轻量，适合中小项目。我们三个人都熟悉 Python，学习成本最低。从零到上线只用了几个星期。

**Q7：SQLite 能支持多少人同时用？**
> 答：SQLite 的 WAL 模式支持一写多读。我们测试过同时 20-30 个请求没问题。学院 1257 人不会同时在线，实际并发很小。如果真的不够，换 MySQL 改一行配置就行。

**Q8：为什么不用 Django？Django 自带后台管理。**
> 答：Django 太重了，自带的功能我们大部分用不上。Flask 更灵活，想要什么自己加，代码更简洁。我们的管理员界面也是根据实际需求定制的，比 Django 自动生成的好用。

**Q9：前端为什么不用 Bootstrap？**
> 答：Bootstrap 引入了很多我们用不到的样式和 JS，文件大。我们自己写的 CSS 只有 10KB，完全贴合移动端需求，没有多余代码。

**Q10：有没有用过 AI 辅助开发？**
> 答：用了 Claude Code 辅助编码和调试，主要是加快开发速度、检查代码问题。但核心设计和逻辑是我们自己完成的。

## 功能实现类

**Q11：学分是怎么计算的？**
> 答：根据竞赛级别（超级/一级/二级/三级）、获奖层次（国家级/省级/校级）、获奖等级（一等奖/二等奖/三等奖）、是否队长四个参数综合计算。计算公式参考了学院学科竞赛学分认定标准。系统在申报提交时自动计算并存储到数据库。

**Q12：如果学生选错了竞赛，怎么修改？**
> 答：当前设计是提交后不能自己修改，防止审核中途数据变动。可以联系管理员驳回，学生重新提交。这是出于数据一致性的考虑。

**Q13：驳回后能不能重新提交？**
> 答：当前版本是驳回后该条申报状态变为已驳回，学生需要重新申报一条新的。后续可以优化为"驳回后可编辑重新提交"。

**Q14：证书图片存在哪里？能存多久？**
> 答：存在 PythonAnywhere 的文件系统里，uploads/certificates 目录下。文件名用 UUID 随机生成，不会重名。PythonAnywhere 免费计划有 3GB 磁盘，按每张 2MB 算能存 1500 张。后续可以迁移到云存储。

**Q15：导出的 Excel 能在手机上打开吗？**
> 答：能。生成的 xlsx 是标准格式，WPS Office 和手机版 Excel 都能打开。合并单元格、表头颜色在手机上也能正常显示。

## 安全类

**Q16：怎么防止有人冒用别人学号登录？**
> 答：初始密码是学号后 6 位或 123456，但首次登录强制修改密码，改完才能用。如果连续输错密码没有限制——这个问题我们知道，后续可以加登录失败次数限制和验证码。

**Q17：怎么防止学生伪造证书图片？**
> 答：系统不能自动检测图片是否伪造。管理员审核时需要对照竞赛目录核实竞赛名称、获奖等级和证书内容。证书原件由学院教务科保留备查。技术层面可以加图片 EXIF 分析和重复图片检测，但最终还是要靠人工审核。

**Q18：系统被攻击怎么办？**
> 答：我们已经做了基本的防护——SQL 参数化查询防注入、密码哈希存储、Session 签名防篡改。Flask 本身也有 CSRF 保护。作为内网系统我们没有暴露敏感端口，PythonAnywhere 也有基础的防火墙。

## 项目管理类

**Q19：你们三个人怎么协作的？有没有遇到冲突？**
> 答：通过 GitHub 管理代码。每个人在自己的分支上开发，完成后提交 Pull Request，另外两个人审查后合并。遇到过代码冲突，用 git merge 解决。整个过程还算顺利。

**Q20：开发过程中最大的困难是什么？**
> 答：一个是 Excel 导出的单元格合并，openpyxl 的 API 要逐行控制合并范围，逻辑比较绕。另一个是 Canvas 图表的标签自适应，不同屏幕、不同数据量情况下字号和位置的平衡调试了很多次。

**Q21：如果重新做一次，会有什么不一样？**
> 答：可能会一开始就引入一些自动化测试，减少人工回归测试的时间。前端可能会用 Vue.js 做组件化，页面多了以后原生 JS 维护起来会比较麻烦。但就目前的规模来说，现在的方案是合理的。

## 老师可能追问的刁钻问题

**刁钻1：你这个系统没什么技术含量嘛，不就是增删改查？**
> 答：表面上确实是增删改查，但具体实现中有很多细节——Canvas 原生画图、Excel 单元格合并、多图累积上传、并发冲突保护、名称清理算法、批量导入的专业自动推断。而且一个系统的价值不只在技术复杂度，更在它能不能解决实际问题。我们做到了零经费、能落地、有人在用。

**刁钻2：既然 PythonAnywhere 免费，那是不是不稳定？**
> 答：PythonAnywhere 免费计划确实有限制——每 3 个月要手动续期、CPU 有每日限额。但对于学院内部管理系统来说够用了。如果学院愿意出 5 美元/月升级付费版，这些问题就都没了。

**刁钻3：你们的数据能导出，是不是意味着可以随意拿走学生信息？**
> 答：导出功能只有管理员能用，学生端没有这个功能。管理员的账号是单独管理的，只有教务科老师有权限。

**刁钻4：用 AI 写的代码你们自己能讲清楚吗？**
> 答：每行代码我们都能讲清楚是干什么的。AI 帮我们加快了开发速度，比如调试、找 bug、优化代码，但核心设计——数据库怎么设计、业务流程怎么走、哪些功能是必需的——都是我们自己决定的。
