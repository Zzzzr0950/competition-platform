## 张三：后端与数据库（组长）

### 你负责了什么
数据库设计、Flask 架构搭建、用户认证、竞赛目录管理、管理员审核模块、学分计算、PythonAnywhere 部署。

### 怎么讲（约 4 分钟）

**1. 数据库设计（30秒）**
"我们设计了四张核心表：users 用户表、submissions 申报表、competition_catalog 竞赛目录表、submission_logs 操作日志表。users 和 submissions 是一对多关系，一个学生可以申报多个竞赛；submissions 和 competition_catalog 通过 catalog_id 外键关联。数据库选用 SQLite，开发阶段零配置，部署时也无需单独安装数据库服务。"

**2. Flask 架构（30秒）**
"后端采用 Flask 工厂模式创建应用，通过 Blueprint 蓝图将认证、学生端、管理员端拆分为独立模块。路由层使用装饰器实现角色权限控制——login_required 和 admin_required 两个装饰器，分别校验登录状态和管理员身份。API 返回统一使用 jsonify，前端通过 fetch 调用。"

**3. 管理员审核（1分钟）**
"审核模块是整个系统的核心。管理员进入审核列表后，可以按状态、班级、竞赛名称、学号姓名多条件组合筛选。点击某条申报进入详情页，查看完整的申报信息和证书图片，填写审核意见后通过或驳回。批量审核可以勾选多条一次性处理。每次审核操作都会写入 submission_logs 表，记录操作者、时间、新旧状态和备注，形成完整的审核轨迹。"

**4. 防并发冲突（30秒）**
"我们考虑到多管理员同时审核的场景，在通过和驳回接口中增加了 pending 状态校验。如果一条申报已经被其他人先审了，后面的人点击时会返回 409 冲突提示'该申报已被其他人审核过了，请刷新列表'。这是用乐观锁的思想在应用层实现的。"

**5. 部署（30秒）**
"部署在 PythonAnywhere 云平台，通过 Git 同步代码，修改后 push 到 GitHub，在 PA 上 git pull 然后 Reload 即可生效。SECRET_KEY 持久化到文件，避免了云平台重启导致全部用户 session 失效的问题。"

### 可能被问到的问题

**Q1：为什么选 SQLite 而不是 MySQL？**
> "SQLite 是嵌入式数据库，零配置、零维护，非常适合我们这种中小规模的应用。当前全院 1257 名学生、预估每年申报量在 2000 条左右，SQLite 完全能够胜任。而且 SQLite 的 WAL 模式支持读写并发，日常使用没有问题。如果未来扩展到其他学院，可以迁移到 MySQL，Flask 只需要改一行连接字符串。"

**Q2：密码是怎么加密存储的？**
> "使用 Werkzeug 的 generate_password_hash 函数，底层是 scrypt 算法。scrypt 是一种内存硬化的哈希算法，能有效抵御暴力破解和彩虹表攻击。数据库里只存哈希值，即使数据库泄露也无法还原明文密码。"

**Q3：怎么防止 SQL 注入？**
> "所有数据库查询都使用参数化查询——用 ? 占位符传参，由 SQLite 驱动做转义处理，不会拼接 SQL 字符串。"

**Q4：Session 是怎么管理的？**
> "Flask 内置的 Session 机制，服务端签名、客户端 Cookie 存储。登录时设置 session['user_id'] 和 session['role']，后续请求通过装饰器校验。设置了 24 小时过期时间，且 SECRET_KEY 持久化到文件，不会因服务重启而失效。"

## 李四：前端与交互

### 你负责了什么
CSS 样式系统、学生申报页面、管理员审核页面、用户管理页面、修改密码页面、登录注册页面、移动端适配、toast/loading 组件。

### 怎么讲（约 4 分钟）

**1. 移动端优先设计（1分钟）**
"我们的用户主要是手机访问，所以采用了 Mobile First 设计策略。CSS 用变量系统统一管理颜色、圆角、间距——比如 --color-primary 主色调蓝色、--nav-height 导航栏高度，整个系统视觉一致。按钮和输入框最小高度 48px，符合 iOS 和 Android 的触控规范。底部导航栏只有三个入口，拇指区域操作方便。还适配了 iPhone 的安全区域，避免内容被底部指示条遮挡。"

**2. QQ/微信浏览器兼容（30秒）**
"因为学生主要在 QQ 群和微信群收到链接，我们特别注意了内置浏览器的兼容性。去掉了点击高亮效果、禁止了自动字体缩放、使用了系统字体栈，确保在这些浏览器里显示和操作都正常。"

**3. 学生申报页面（1分钟）**
"申报页面使用了手风琴组件展示竞赛目录，按类别分组、按级别排序。学生可以逐层展开、搜索筛选，点击选中后关闭面板。表单验证分前端和后端两层——前端用 JS 实时校验必填项和高亮错误字段，后端再校验一次确保数据合法性。文件上传支持预览，图片直接显示缩略图，PDF 显示红色图标。"

**4. 管理员审核页面（30秒）**
"审核页面的筛选栏支持状态、班级、竞赛、关键词四个维度组合筛选。列表项用卡片式布局，显示学生信息、竞赛名称、状态标签，待审核的显示通过和驳回按钮。分页每页 15 条，底部显示总数和页数。"

**5. 用户体验细节（30秒）**
"toast 提示消息 4 秒自动消失、loading 遮罩防止重复提交、confirm 弹窗确认删除操作、表单错误字段高亮标红。这些都是用原生 JS 实现的，没有依赖任何框架或组件库。"

### 可能被问到的问题

**Q1：为什么不用前端框架，比如 Vue 或 React？**
> "这个项目的页面数量有限，大约 10 个页面，交互逻辑也不算复杂。引入 Vue 或 React 会增加打包配置、依赖管理、学习成本，而我们选择原生 HTML/CSS/JS，零依赖、加载快、维护简单。移动端打开页面 CSS 和 JS 总共才几十 KB，首屏加载不到 1 秒。对于这个体量的项目，原生方案是完全够用的。"

**Q2：怎么确保不同手机屏幕的适配？**
> "使用了弹性布局 flexbox 和百分比宽度，不使用固定像素值限定宽度。CSS 变量定义了统一的间距和圆角。通过 max-width: 768px 限制内容区宽度，在大屏幕上居中显示还不浪费空间。实际测试过 iPhone SE（320px）、iPhone 14（390px）、主流安卓（360-428px）和 iPad（768-1024px）都正常。"

**Q3：页面加载慢怎么办？**
> "我们的静态资源非常小——CSS 约 10KB、JS 总共约 20KB，没有外部字体、没有图标库（用的 emoji），首屏加载很快。HTML 模板是服务端渲染的，数据通过 fetch API 异步加载，页面结构先出来、数据后填充，用户感知速度很快。"

**Q4：表单提交失败了怎么处理？**
> "前端的 submit 事件先做字段级校验，不合格的直接显示错误提示不发送请求。请求发出后按钮禁用防止重复提交。后端返回错误时 showToast 显示具体原因——'请选择竞赛'、'文件格式不支持'等。网络异常也会提示用户检查网络后重试。"

## 王五：数据统计与工具脚本

### 你负责了什么
Canvas 统计图表引擎、Excel 报表导出、竞赛名称清理算法、多图累积上传、自定义日历、批量导入脚本、强制改密码机制、防并发审核。

### 怎么讲（约 4 分钟）

**1. Canvas 统计图表（1.5分钟）——这是最亮眼的部分**
"统计模块是管理员最常用的功能之一。我们没用 Chart.js 或 ECharts，直接用 HTML5 Canvas 原生 API 画了三张图。按竞赛级别分布的柱状图——国家级紫色、省级蓝色、校级灰色，柱子下面的标签根据文字长度自动调整字号，最长能到 9px，确保不截断。按班级分布的柱状图同理，班级名水平居中显示。最有特色的是按竞赛项目 Top 10 的横向条形图——条形宽度和申报数量成正比，颜色从深蓝到浅蓝渐变，数量越多颜色越深，一眼就能看出哪些竞赛参与热度高。

竞赛名称这块有个细节——目录里有些竞赛带着 ①②③ 这样的圈号数字和子赛项列表，比如'中国高校计算机大赛-①大数据挑战赛、②团体程序设计天梯赛……'，直接显示太长了。我写了一个名称清理函数，用正则匹配 ①-⑳ 的位置，向前找到分隔符就截断，保留主竞赛名称。图表里显示'中国高校计算机大赛'就够了，完整名称在申报选择里能找到。"

**2. Excel 导出（1分钟）**
"导出是管理员最需要的功能。我们用 openpyxl 生成真正的 xlsx 文件，不是传统的 CSV。特色有三个：第一，同一学生的学号、姓名、总学分单元格纵向合并，一个学生多条申报看起来一目了然；第二，蓝色表头、居中对齐、冻结首行、列宽自适应，打开就能用不用调格式；第三，用的是 LEFT JOIN 查询，选了某个班导出，这个班所有人都出现在表里——包括一次都没申报的同学，学号姓名照常显示，竞赛栏位留空。这就是班级花名册式的完整报表。"

**3. 批量导入（1分钟）**
"我们从教务系统拿到了全院的 Excel 名单。写了导入脚本，支持直接读 xlsx 文件，自动识别学号、姓名、班级、专业列，模糊匹配表头。学院班级编号有规则——8 位班级号的第 6 位代表专业，1 是计算机科学与技术、2 是物联网工程、3 是数据科学与大数据技术。脚本自动解析这个规则填充专业字段，不需要手动标注。还可以选择跳过某个年级。导入后每个学生的 must_change_password 标记为 1，首次登录强制跳转改密码页，初始密码统一 123456 用过一次就失效。"

**4. 多图上传 + 日历（30秒）**
"多图上传解决了选择文件时后面覆盖前面的问题——每次选择追加到内存队列，不替换。日历组件是自己写的，点击日期框弹出居中面板，年月可以快速切换，日期按钮大，比手机原生的好用。"

### 可能被问到的问题

**Q1：为什么不用 ECharts，要自己画？**
> "ECharts 是一个 1MB+ 的库，加载慢、依赖多。我们的需求就是柱状图和条形图，用 Canvas 原生 API 写只需要大约 180 行代码，零依赖、文件体积几 KB。而且自己画的话，像标签自适应字号、颜色线性渐变、动态左间距这些细节都能完全定制。对于数据量不大的统计场景，这是一个很合理的技术选择。"

**Q2：Canvas 图表在手机上显示正常吗？**
> "正常的。Canvas 最小宽度 600px，在手机上如果容器不够宽就会出现水平滚动条，条形图的竞赛名称和柱子都能完整看到。柱状图的标签字号会根据柱子宽度自动计算，最多缩到 9px，不会再小。实际测试在 iPhone SE 到 iPad Pro 上都正常。"

**Q3：Excel 导出的单元格合并怎么实现的？**
> "导出时先按学号分组，遍历每个学生的申报列表。第一个申报写完整的学号、姓名、总学分；同一学生的后续申报这些格子留空。然后用 openpyxl 的 merge_cells 方法，把学号列、姓名列、总学分列的开始行到结束行进行单元格合并。这样在 Excel 里打开就是合并居中显示的效果。"

**Q4：导入时怎么处理重复的学号？**
> "导入前先查询数据库，如果学号已存在就跳过，不会覆盖已有数据。所以这个脚本可以重复跑，只导入新增的学生，已有的不受影响。"

**Q5：名称清理算法是怎么工作的？**
> "核心是一个 JavaScript 正则 `/[①-⑳]/`，搜索圈号数字在字符串中的位置。找到后从那个位置往前扫描，寻找分隔符——中文破折号、英文减号、中英文冒号、逗号、顿号、空格等。找到分隔符就从那里截断，返回截断前的部分。如果没找到圈号数字就直接返回原名。这样处理之后，'中国高校计算机大赛-①大数据挑战赛、②……'就变成'中国高校计算机大赛'，清爽很多。"

## 答辩通用 Tips

**演示环节**：
- 提前打开 3-4 个关键页面（管理首页、统计页、审核页、学生申报页），切标签就能展示
- 提前登录好，不要当场输密码
- 录一段 2 分钟操作视频做备用，万一网络挂了就播视频

**回答问题原则**：
- 不确定的不要说"不知道"，说"这个我们考虑过，目前的方案是……后续可以……"
- 技术选型的问题统一回答"中小规模、零依赖、够用就好"
- 被问数据安全就说 scrypt 加密、参数化查询、session 签名
- 被问推广就说"替换竞赛目录和学生数据即可"

**三个人之间的衔接**：
- 张三开头："我们开发了一套竞赛获奖申报管理平台……下面请李四介绍前端设计"
- 李四："前端采用了移动端优先策略……下面请王五介绍数据统计和工具"
- 王五收尾："以上是我们团队的全部工作，请老师们提问"
