# 竞享成果——学生竞赛获奖申报平台 结题报告

---

## 一、项目实施工作概况总结

### 1.1 项目背景与确立

中北大学计算机科学与技术学院高度重视学生学科竞赛工作，每年组织学生参加各级各类学科竞赛百余项，涵盖中国国际大学生创新大赛、"挑战杯"系列竞赛、ACM-ICPC 国际大学生程序设计竞赛、全国大学生数学建模竞赛、中国大学生计算机设计大赛等知名赛事。随着学科竞赛规模和学生参与度的持续扩大，传统的线下纸质申报管理方式暴露出诸多问题：

**申报环节痛点**：学生需从学院官网下载竞赛目录，手动查找对应竞赛编号，填写纸质申报表，复印获奖证书，在规定时间内提交至学院教务办公室。流程繁琐、耗时长，尤其对于在外实习或出差参赛的学生极为不便。部分学生因未及时看到通知而错过申报截止日期。

**审核环节痛点**：管理员需逐份翻阅纸质申报材料，核对竞赛目录、获奖等级、证书真伪等信息。同一竞赛可能有数十名学生同时申报，手工汇总统计易出错。审核意见和结果无法实时反馈给学生，学生需多次往返办公室询问审核进度。

**统计归档痛点**：每学年末需按班级、竞赛级别、获奖等级等维度汇总统计全院竞赛成果，手工制作 Excel 报表耗时数天。数据分散在纸质表格中，历年数据对比分析困难，不利于学院学科竞赛工作的长期规划和决策。

经学院领导讨论决定，开发一套基于 Web 的竞赛获奖申报管理平台，实现竞赛申报全流程数字化管理，全面提升学院学科竞赛管理工作的效率与规范化水平。

### 1.2 项目总体规划与设计思路

本系统采用 B/S（Browser/Server）架构模式，以 Python Flask 轻量级 Web 框架为核心，SQLite 轻量级数据库为数据存储引擎，前端采用移动端优先的响应式设计理念，确保系统可在手机（含 QQ 浏览器、微信内置浏览器）、平板电脑、桌面 PC 等多种终端上流畅运行。

系统整体划分为两个核心功能模块：

**（1）学生端模块**：面向全院本科生，提供在线申报竞赛获奖的全流程服务。学生登录后可浏览 2026 年学科竞赛完整目录，支持按类别、级别、关键词搜索竞赛；选择竞赛后填写获奖等级、获奖层次（国家级/省级/校级）、获奖日期、团队信息等申报信息，上传获奖证书图片作为证明材料；提交后实时追踪审核状态（审核中/已通过/已驳回）；审核通过后系统自动计算竞赛学分并累加总学分。学生可在个人中心查看历次申报记录，支持按状态筛选。

**（2）管理员端模块**：面向学院教务管理人员，提供审核管理、数据统计、报表导出、系统配置等综合管理功能。管理员可查看待审核申报列表，支持按状态、班级、竞赛等多条件组合筛选；逐条审核申报材料，在线查看证书图片，填写审核意见后通过或驳回，支持批量审核操作；审核结果自动记录操作日志，支持多管理员并发审核且具备冲突检测保护。数据统计模块以可视化图表展示各级别竞赛分布、各竞赛项目申报热度、各班级参与情况等关键指标。报表导出功能支持按班级筛选，生成规范格式的 Excel 文件，包含全班所有学生信息（含未申报学生），自动计算每人总学分，同一学生的学号和姓名单元格自动合并居中。

### 1.3 前期准备工作

项目启动前，课题组成员系统性地完成了以下准备工作：

**（1）需求调研与分析**：深入走访学院教务科、团委、学生工作办公室等部门，与教务管理员、辅导员、各年级学生代表进行座谈，全面了解竞赛申报管理的现存问题和各方需求。梳理出核心需求清单 30 余项，按优先级分为 P0（必须有）、P1（应该有）、P2（最好有）三个等级，为后续迭代开发提供依据。

**（2）竞赛目录整理**：收集并整理中北大学 2026 年学科竞赛完整目录，涵盖超级、一级、二级、三级四个级别共 125 项赛事，涉及创新创业、学科竞赛两大类别。每项赛事标注了完整名称、简称、主办单位、归属学院、推荐级别等信息，数据来源于学院教务科官方发布的《中北大学学科竞赛目录（2026 版）》，确保与学校政策完全一致。

**（3）学生数据准备**：从教务系统导出全院三个年级（2023 级、2024 级、2025 级）27 个班级共 1257 名本科生的学号、姓名、班级信息。按照班级编号第 6 位编码规则（1=计算机科学与技术、2=物联网工程、3=数据科学与大数据技术），自动为每位学生标注专业方向。

**（4）技术选型与架构设计**：对比 Django、Flask、FastAPI 等主流 Python Web 框架后，选择 Flask 作为开发框架，因其轻量灵活、学习成本低、文档丰富。数据库选用 SQLite，满足中小规模应用需求且无需单独部署数据库服务。前端采用原生 HTML/CSS/JavaScript，不引入重量级框架，确保移动端加载速度快、兼容性好。部署平台选用 PythonAnywhere，提供免费云托管服务，支持 Git 自动部署。

**（5）开发环境搭建**：配置 Python 虚拟环境、安装依赖库、初始化 Git 仓库并关联 GitHub 远程仓库。建立项目目录结构规范，编写 .gitignore 规则，确保代码管理规范有序。

**（6）项目管理规划**：制定项目甘特图与里程碑节点，明确各阶段交付物和验收标准。采用 GitHub Issues 进行任务跟踪，每次提交关联具体功能或修复。

### 1.4 项目实施过程管理

项目实施严格遵循软件开发流程规范，采用敏捷迭代开发模式，分为五个阶段推进：

**第一阶段——基础框架搭建（第 1-2 周）**：完成 Flask 项目初始化、数据库表结构设计与创建（users、submissions、competition_catalog、submission_logs 四张核心表）、用户认证模块（登录、注册、Session 管理、角色权限控制）、基础页面模板与 CSS 样式系统搭建。此阶段交付物为可运行的项目骨架，具备基本的登录和角色区分能力。

**第二阶段——核心业务实现（第 3-4 周）**：完成竞赛目录管理（添加、编辑、删除、搜索竞赛）、学生申报表单（竞赛选择手风琴组件、获奖信息填写、单张证书上传）、管理员审核流程（审核列表、审核详情、通过/驳回操作）。此阶段交付物为具备完整申报-审核闭环的可用系统。

**第三阶段——数据统计与导出（第 5-6 周）**：完成数据统计图表模块（Canvas 原生柱状图、横向条形图，支持按级别、竞赛、班级三个维度展示）、CSV 报表导出功能、分页与筛选优化。此阶段交付物为具备数据可视化和报表导出能力的增强版系统。

**第四阶段——批量管理与系统优化（第 7-8 周）**：完成学生批量导入脚本（支持 Excel 直接读取、自动推断专业、跳过指定年级）、首次登录强制改密码机制、公开注册开关、多管理员并发审核冲突保护、审核页面班级筛选、竞赛名称智能清理算法。此阶段交付物为接近生产就绪的完整系统。

**第五阶段——用户体验打磨（第 9-10 周）**：完成多图累积上传机制、自定义日历日期选择器、统计图表标签优化（水平显示、自适应字号、动态左间距）、Excel 替代 CSV（单元格合并、蓝色表头、居中排版）、Session 持久化修复、禁止浏览器缓存、已登录用户自动跳转等细节优化。此阶段交付物为正式上线运行的最终版本。

## 二、项目主要技术特征和创新点

### 2.1 主要技术特征

#### 2.1.1 移动端优先的响应式设计体系

系统前端设计采用"Mobile First"（移动端优先）策略，确保在手机屏幕（320px-428px）上获得最佳操作体验，同时向上兼容平板（768px+）和桌面端（1024px+）。具体技术实现包括：

- **CSS 变量主题系统**：定义 `--color-primary`、`--color-success`、`--nav-height`、`--bottom-nav-height` 等全局 CSS 变量，统一管理颜色、间距、圆角等视觉参数，便于主题切换和维护。
- **弹性布局与最小触摸区域**：按钮和交互元素最小高度 48px，确保符合 iOS 和 Android 人机交互规范。表单输入框宽度始终 100%，避免移动端溢出。
- **底部导航栏**：借鉴移动 App 设计模式，学生端采用三栏底部固定导航（首页、申报获奖、我的申报），管理员端采用三栏底部导航（管理首页、审核申报、数据统计），拇指热区操作便捷。
- **安全区域适配**：通过 `env(safe-area-inset-bottom)` 适配 iPhone X 及后续机型的底部安全区域，避免内容被 Home Indicator 遮挡。
- **QQ/微信浏览器兼容**：使用 `-webkit-tap-highlight-color: transparent` 消除点击高亮，`-webkit-text-size-adjust: 100%` 防止字体缩放，确保在 QQ 和微信内置浏览器中显示正常。

#### 2.1.2 原生 Canvas 数据可视化引擎

为保持项目零外部依赖的轻量特性，统计图表模块完全基于 HTML5 Canvas 原生 API 实现，未引入 Chart.js、ECharts 等第三方图表库。该引擎具备以下能力：

- **垂直柱状图**：用于"按竞赛级别分布"和"按班级分布"统计。支持网格线绘制、Y 轴刻度自动计算、标签自适应字号（班级名越长字号越小，最小 9px，确保标签完整显示在柱子下方）。颜色支持自定义映射（如国家级=紫色、省级=蓝色、校级=灰色）。
- **横向条形图**：用于"按竞赛项目 Top 10"统计。标签区域宽度根据最长竞赛名称的 `measureText()` 实际像素值动态计算，超长名称自动在自然断点（括号、顿号、空格等）处换两行显示。条形宽度与申报数量成正比，颜色通过 RGB 线性插值实现深浅渐变——数量越多颜色越深（深蓝 #1a56db），数量越少颜色越浅（浅蓝 #60a5fa）。
- **自适应布局**：Canvas 最小宽度 600px，窄屏设备上容器启用 `overflow-x: auto` 水平滚动，确保标签和条形均完整可见。
- **空数据与异常处理**：API 请求失败时显示错误图标和提示文字；无数据时显示"暂无数据"占位文本；Canvas 不支持时输出友好提示。

#### 2.1.3 多图累积上传机制

传统 HTML 文件选择器每次打开都会替换之前已选的文件，无法实现分批追加选择。本系统开发了基于 JavaScript 内存队列的累积上传方案：

- **文件队列维护**：全局数组 `selectedFiles[]` 持续保存用户通过拍照、相册、拖放三种方式添加的所有文件。每次新选择执行 `push` 追加操作，而非替换。
- **可视化预览管理**：`refreshPreviews()` 函数遍历完整文件队列，为每张图片生成缩略图预览（80×60px），PDF 文件显示红色标识。每张预览右上角有删除按钮，点击仅移除对应索引的文件，其余不受影响。
- **表单提交适配**：提交时遍历 `selectedFiles[]`，逐个通过 `formData.append('certificate', file)` 添加到 FormData，后端 `request.files.getlist('certificate')` 接收全部文件，逗号分隔存储路径。
- **向后兼容**：后端 `parse_certificate_images()` 函数兼容旧版单张路径和新版逗号分隔多路径两种格式，已有数据无需迁移。

#### 2.1.4 Excel 格式导出与单元格合并

传统 CSV 格式不支持单元格合并、样式设置和列宽调整。本系统采用 openpyxl 库生成符合 Office Open XML 标准的 .xlsx 文件，实现以下增强：

- **单元格纵向合并**：同一学生的多条申报记录，学号列、姓名列、总学分列自动纵向合并为一个单元格，视觉清晰规范。
- **样式设置**：表头使用深蓝底色（#1A56DB）+ 白色加粗字体；所有数据单元格添加淡灰色边框；内容水平居中、垂直居中（竞赛名称和审核意见列左对齐以适配长文本）。
- **自适应列宽**：14 列分别预设合理宽度（学号 16、姓名 10、竞赛名称 36、级别 10 等），确保打开即可阅读，无需手动调整。
- **表头冻结**：首行冻结，滚动查看数据时表头始终可见。
- **全班覆盖导出**：导出 SQL 查询使用 `FROM users LEFT JOIN submissions`，确保所选班级的每一位学生都出现在报表中（含零申报学生），学号姓名正常显示，竞赛栏位留空。按学号升序排列。

#### 2.1.5 Session 安全与并发控制

- **SECRET_KEY 持久化**：原代码使用 `secrets.token_hex(32)` 运行时随机生成 Flask 密钥，导致 PythonAnywhere WSGI 进程重载后所有用户 Session 失效。修改为优先读取环境变量，其次读取 `.secret_key` 持久化文件，均不存在时才生成并写入文件，确保密钥跨重启稳定。
- **防并发审核冲突**：审批通过和驳回接口增加 `status != 'pending'` 前置校验，若申报已被其他管理员先行审核，返回 HTTP 409 Conflict 并提示"该申报已被其他人审核过了，请刷新列表"。批量审核在 SQL 查询层面已包含 `AND status = 'pending'` 条件，自动跳过已审核项。
- **禁止浏览器缓存**：全局 `after_request` 中间件添加 `Cache-Control: no-store, no-cache, must-revalidate` 响应头，防止手机浏览器缓存导致会话状态显示混乱。
- **已登录自动跳转**：访问 `/login` 路由时检测 Session 中已有 user_id，则直接重定向至对应端首页，避免出现"已登录导航栏 + 登录表单"的混乱界面。

### 2.2 创新点

**（1）竞赛名称智能清理算法**：学科竞赛目录中部分赛事名称附加了详细的子赛项列表，如"中国高校计算机大赛-①大数据挑战赛、②团体程序设计天梯赛、③移动应用创新赛、④网络技术挑战赛、⑤人工智能创意赛"，统计图表中直接显示会使标签过长。本系统开发了基于正则表达式的名称清理函数 `cleanCompetitionName()`，自动检测 ①-⑳（Unicode 圈号数字）字符位置，向前查找分隔符（—、-、：、,、等），从分隔符处截断保留主竞赛名称，无圈号数字的名称保持原样。该算法确保统计图表标签简洁可读，同时完整的竞赛名称在申报选择列表中保持不变，不影响学生正常选择子赛项。

**（2）班级编码专业自动推断**：学院班级编号遵循固定编码规则，8 位班级号的第 6 位数字对应不同专业方向（1=计算机科学与技术、2=物联网工程、3=数据科学与大数据技术）。批量导入脚本利用此规则，在 Excel 表格缺少专业列或专业字段为空时，自动解析班级号第 6 位填充专业信息，减少数据准备工作量。该规则通过 `guess_major()` 函数实现，支持后续规则扩展。

**（3）首次登录强制改密码机制**：数据库 `users` 表新增 `must_change_password` 标志字段，批量导入学生时默认设为 1。登录成功后优先检查该标志，若为真则跳过正常页面跳转，直接重定向至修改密码页并显示提示信息。修改密码成功后自动清除标志位，后续登录正常进入系统。该机制确保了批量导入的初始密码（统一 123456）在学生首次使用后即被替换，保障账号安全。

**（4）自定义日历日期选择器**：移动端不同浏览器的原生 `<input type="date">` 日期控件表现不一致，部分浏览器日历弹窗过小、操作不便。本系统开发了基于弹窗遮罩的自定义日历组件，包含年份网格快速跳转（以 10 年为区间展示 12 个年份按钮）、月份网格点击切换（4 列 3 行）、日期网格（7 列，支持今天高亮）。日历居中弹窗展示（max-width 340px），半透明遮罩防止误触，点击遮罩外部或取消按钮关闭。该组件完全由原生 JavaScript 和 CSS 实现，无外部依赖。

**（5）LEFT JOIN 全覆盖导出策略**：传统竞赛统计导出仅展示有申报记录的学生，导致导出的班级报表人数不完整。本系统导出采用 `FROM users LEFT JOIN submissions` 查询策略，以学生表为主表，左连接申报表和竞赛目录表，确保全班所有学生出现在导出结果中。无申报记录的学生所有竞赛相关字段为 NULL，在 Excel 中显示为空白单元格，但学号、姓名、总学分（0）正常显示，实现了班级花名册式的完整统计报表。

## 六、存在的问题及对策措施

### 6.1 存在的问题

#### （1）SQLite 数据库并发性能局限

SQLite 为嵌入式文件型数据库引擎，采用库级写锁机制，同一时刻只允许一个写操作。当同时在线审核的管理员较多，或多个学生同时提交申报时，可能出现"database is locked"锁等待错误。经测试，并发写入超过 10-15 个时性能开始下降。该问题属于 SQLite 架构固有特性，非代码层面可完全解决。

#### （2）PythonAnywhere 免费计划资源受限

当前部署的 PythonAnywhere Beginner 免费计划具有以下限制：磁盘空间 3GB（证书图片长期积累有超限风险）、每日 CPU 处理时间限制（高峰期可能触发限流）、不支持 HTTPS 自定义证书（访问地址为 HTTP 协议）。此外，免费计划每 3 个月需手动续期一次，若忘记续期网站将停摆。

#### （3）证书图片未经压缩处理

当前上传的证书图片直接以原始分辨率保存，部分手机拍摄的照片单张可达 5-10MB，不仅占用磁盘空间，也导致审核详情页加载缓慢。系统仅进行了缩略图生成（400px 宽用于预览），但原图未做尺寸和质量压缩。

#### （4）缺乏消息通知机制

学生提交申报后，审核结果（通过/驳回）无主动通知渠道。学生需自行登录系统查看审核状态，若忘记查看可能错过后续处理（如被驳回后需修改重新提交）。驳回时管理员的审核意见也无法触达学生。

#### （5）操作审计日志不完整

当前系统仅对申报审核操作（通过/驳回）记录了 `submission_logs` 审计日志。管理员的竞赛目录增删改、用户账户冻结/解冻、密码重置等敏感操作未记录完整审计轨迹，不利于事后追溯和责任认定。

#### （6）系统可维护性有待提升

项目初期追求快速开发和零依赖，后端路由文件（admin.py）代码量较大（400+ 行），部分业务逻辑与路由处理耦合，后续功能扩展时维护成本可能升高。前端 JavaScript 文件缺乏模块化打包工具支持，代码组织依赖手动管理。

#### （7）缺少自动化测试覆盖

项目未引入单元测试或端到端测试框架，功能验证依赖人工测试。随着系统功能持续增加，回归测试工作量增大，自动化测试的缺失可能导致新功能引入隐性缺陷。

### 6.2 对策措施

#### （1）数据库升级至 MySQL/PostgreSQL

当系统用户规模和服务需求增长到一定程度后，可将数据库从 SQLite 迁移至 MySQL 或 PostgreSQL 等 C/S 架构的关系型数据库。升级后可获得行级锁、连接池、查询缓存、主从复制等企业级特性，并发写入能力可提升至数百个连接。PythonAnywhere 付费计划（$5/月）已内置 MySQL 支持。迁移成本较低——Flask 配合 SQLAlchemy ORM 替换裸 SQL 查询，数据库方言切换仅需修改连接字符串。

#### （2）对象存储迁移与 HTTPS 升级

证书图片迁移至阿里云 OSS 或腾讯云 COS 对象存储服务（均有免费额度），解决文件存储空间和带宽问题。前端直接通过 CDN 加速 URL 访问图片，减少服务器负载。将 PythonAnywhere 计划升级至付费版（$5/月），开启 HTTPS 支持，提升安全性和用户体验。

#### （3）前端压缩 + 后端压缩双重图片优化

在前端文件选择后、上传前增加 Canvas 压缩步骤：将图片绘制到限定尺寸（如 1920px 宽）的离屏 Canvas 上，导出为 JPEG（quality 0.7），文件大小可压缩至原图的 10%-20%。后端 `save_upload()` 函数在保存后调用 Pillow 进行二次压缩：生成最大宽度 1200px 的优化版本作为审核查看图，仅保留原图作为归档备查。预计整体存储空间节约 60% 以上。

#### （4）接入消息推送渠道

对接微信公众号模板消息 API 或企业微信机器人 Webhook，在审核状态变更时向学生推送通知："您的 [竞赛名称] 申报已通过/驳回"。驳回通知中附带管理员审核意见，学生点击可跳转至系统修改重新提交。该方案需申请微信公众号或企业微信开放平台账号（免费），开发工作量约 2-3 天。

#### （5）完善操作审计日志

扩展 `submission_logs` 表或新建 `admin_audit_logs` 表，对所有敏感操作记录审计信息：操作时间、操作者 ID、IP 地址、操作类型（如"添加竞赛""删除竞赛""冻结用户""重置密码"等）、操作对象 ID、操作前值、操作后值、备注。为管理员用户管理、竞赛目录管理页面增加"操作日志"查看入口，支持按时间、操作类型、操作者筛选。

#### （6）后端模块化重构 + 前端工程化

对 admin.py 和 student.py 进行拆分，将数据查询逻辑抽取到独立的 service 层（如 `services/review_service.py`、`services/export_service.py`），路由层仅负责请求参数校验和响应格式化。前端引入简单的构建流程（如使用 esbuild 打包 JavaScript、PostCSS 处理 CSS），使代码组织更清晰、加载更快。不引入重量级框架的情况下保持项目轻量特性。

#### （7）引入自动化测试

使用 pytest + pytest-flask 编写后端 API 单元测试，覆盖认证、申报、审核、导出等核心流程。前端使用简单的 HTML 页面测试清单，或引入 Playwright 进行端到端自动化测试。测试集成至 GitHub Actions，每次 push 自动运行，确保新提交不引入回归 bug。
